# Logs and replays

Use this guide after the first workflow run when you need to inspect outputs, read step logs, or replay a run to narrow down what happened.

## What this page covers

This launch guide explains where post-run inspection starts and how the existing replay-oriented workflow model helps reviewers understand successful and failed runs without reading source code first.

## Start with the run summary

After a workflow completes, first confirm whether the run reached a clear success state and which steps produced the final result. The run summary is the fastest way to answer three basic questions:

1. Did the workflow finish successfully, stop for approval, or fail?
2. Which step produced the output you expected to review?
3. Which step should you inspect next if the result is missing or incomplete?

Treat the summary as the index for the rest of your inspection. If it shows the expected terminal state and a final output-producing step, move there before reading every log line.

## Inspect outputs before deep debugging

For a successful first run, inspect the user-visible outputs before you start replaying or troubleshooting internals. Reviewers should usually look for:

- the final artifact, file change, or generated response the workflow was meant to produce
- any short result summary that explains what the workflow completed
- step outputs that confirm handoffs between agents, tools, and approval points

This keeps the inspection path practical. A good first question is not "what did every step log?" but "what did this run leave behind that proves the workflow did the intended work?"

## Confirm the success signals match the emitted result

A healthy successful run should leave behind more than a green status badge. Before you move on, make sure the completion signal matches the output you inspected:

- the run reached a terminal success state instead of stopping at an approval or retry boundary
- the expected artifact, change, or response is present where that workflow normally emits it
- the result summary and the output-producing step describe the same outcome in plain language

If one of those clues is missing, treat the run as incomplete even if the top-level status looks successful. That usually means the next useful step is to inspect the output-producing step directly instead of assuming the run is done.

## Use step outputs to connect result to workflow behavior

Once you find the final result, step back through the output-producing steps in reverse order. This helps you connect the finished artifact to the workflow behavior that created it.

Look for concrete clues such as:

- the step where the final artifact was emitted or written
- any earlier step that prepared inputs for that output
- approval boundaries that determined whether the workflow could continue
- short status notes that show a step completed with the expected inputs and outputs

If those clues line up, you usually have enough evidence that the run behaved correctly without turning the guide into a full step-by-step trace review.

## Success path checklist

After a successful workflow run, inspect in this order:

1. the overall run status
2. the final output or artifact
3. the step that emitted that result
4. the immediately preceding handoff or approval step
5. only then the broader logs if something still looks unclear

This order keeps the first-review experience grounded in observable results instead of forcing a new user to infer success from raw logs alone.

## When a run fails or stalls, inspect the first failing clue before everything else

When a workflow does not finish cleanly, do not start by reading the entire trace from top to bottom. Start with the first clue that tells you where the run stopped changing state:

- the step marked failed, blocked, or waiting for approval
- the last step that still produced a clear output
- any short error summary, retry note, or approval status attached to the run

Those clues tell you where to focus. A failed run is usually easiest to understand when you compare the last healthy step with the first step that stopped, retried, or asked for human input.

## Read logs for decision points, not just raw volume

Step logs are most useful when they explain a decision boundary. For a confusing first run, inspect logs in this order:

1. the failing or blocked step summary
2. the input or handoff that step received
3. the log line or status note that explains why the step failed, retried, or paused
4. the previous successful step, to confirm whether the problem started earlier

This keeps log review practical. You are looking for signals such as missing inputs, invalid configuration, approval pauses, or tool failures, not for every message the run emitted.

## Use replay or rerun surfaces to narrow the problem

Replay or rerun surfaces are most useful after you have identified the first suspicious step. Use them to answer a narrower question than "does the whole workflow work?":

- replay the run when you need to inspect the same path again and confirm where behavior diverges
- rerun after a fix when you need to verify whether a changed input, approval, or configuration resolves the issue
- compare the repeated run with the original run summary, outputs, and failing step so you can tell what actually changed

The goal is not just to produce a second result. The goal is to prove whether the same step fails again, whether the failure moved, or whether the expected output now appears.

## Debug the first missing-output or approval-bottleneck case

Two common first-adoption problems are a missing expected output and a workflow that appears stuck waiting for approval.

If the output is missing, inspect:

1. whether the run ended in success, failure, or a non-terminal waiting state
2. the step that should have emitted the result
3. the immediately preceding handoff or tool step for missing input or error clues
4. a replay or rerun after fixing the missing dependency, prompt, or configuration

If the run is waiting for approval, inspect:

1. which approval step is blocking progress
2. what output or summary was produced for the approver
3. whether the next step ever started
4. a replay or rerun after the approval path is completed or corrected

In both cases, the debugging path starts from the missing observable outcome and works backward to the first blocked or failing step.

## Failure path checklist

When a workflow fails, pauses unexpectedly, or produces an incomplete result, inspect in this order:

1. the run state and the first failing, blocked, or waiting step
2. the last confirmed output before the problem
3. the logs around that step's decision or error boundary
4. the replay or rerun result after your smallest corrective change
5. whether the expected output now appears or the failure moved somewhere else
