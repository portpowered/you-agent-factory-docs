## Codebase Patterns
- Canonical content types and validation live in `src/lib/content/`; import from `@/lib/content`. Keep this separate from docs-shell UI copy in `src/lib/shell.ts` (re-exported from `@/lib/shared-shell-config`).
- `validateContentMetadata()` returns `{ ok: true, record }` or `{ ok: false, errors }` with reviewer-verifiable `{ field, message }` entries instead of throwing.
- Canonical ids use `{kind}/{slug}` and must match the declared `kind` and `slug`; route paths are derived per public content kind.
- Locale tags use BCP 47-style `en` or `en-US`; `canonicalLocale` must be included in `availableLocales`.
- Root `Makefile` targets (`setup`, `check`, `test`, `build`) mirror `package.json` scripts; prefix `typecheck` with `next typegen` when `.next/types` is referenced.
- Component tests use `@testing-library/react` with happy-dom preloaded from `bunfig.toml`; mock `next/link` via `tests/helpers/mock-next-link.tsx`.
- Starter content fixtures live under `src/content/{docs,blog,glossary,comparisons,references}/{slug}/{locale}.mdx`; invalid fixtures for tests live under `tests/fixtures/starter-content/`.
- Docs-shell navigation is projected via `projectDocsShellNavigation()` / `loadDocsShellNavigation()` into `DocsShellNavigationInput`; `DocsShell` composes `SharedShell` with `projectSharedShellDocsNavigation()` instead of hand-maintained nav constants.
- `requireStarterContentRecords()` and `loadDocsShellNavigation()` throw `StarterContentValidationError` when any starter fixture is invalid; do not silently drop failures on generation paths.
- Published doc pages are served from `src/app/docs/[slug]/page.tsx` via `loadDocPage()` and `generateStaticParams()`.
- Cross-layer foundation verification lives in `tests/unit/canonical-content-foundation.test.ts`; static-export link following belongs in `tests/unit/static-export.test.ts`.
- Starter locale files may use `.md` or `.mdx`; `resolveLocaleFileName()` in `src/lib/content/locale-files.ts` is the shared resolver and `loadDocPage()` prefers `.mdx` when both exist.
- `DocsShell` projects canonical navigation into `createSharedShellConfigFromMessages()` so localized shell labels and generated docs sidebar items compose without hand-maintained nav constants.
- Static-export/browser tests use `getTestPort()` with env overrides so nested `make test` from `root-command-path.test.ts` does not bind the same ports as the outer suite.
- Localized variant identity is projected separately from `CanonicalContentRecord` via `LocalizedContentVariantIdentity` and `validateLocalizedVariantBindings()` in `src/lib/content/localized-variant-identity.ts`; group parallel locale files with `buildStarterContentPathKey()`.
- `loadStarterContentRecords()` deduplicates locale files per slug through `resolveLocaleFileName()` before running localized variant identity validation across bindings.
- Starter locale files require explicit `canonicalLocale`, `availableLocales`, and `id` frontmatter; `validateExplicitStarterLocaleMetadata()` and `validateLocaleRegistryMetadata()` run before canonical record projection in `validateStarterContent()`.
- Group validation fails when the canonical-locale variant file is missing or when `availableLocales` lists locales without matching on-disk variant files.
- `loadStarterContentRecords()` returns `localizedVariantGroups` and `variantBindings` for reviewer-visible parallel variant proof and locale-aware navigation dedupe.
- `projectDocsShellNavigation(records, { variantBindings })` selects the canonical-locale variant label by default; pass `{ locale }` for localized nav labels.
- `resolveLocalizedContentVariant()` and `selectLocalizedVariantBinding()` in `src/lib/content/localized-content-resolution.ts` centralize locale-aware content resolution and canonical-locale fallback; `loadDocPage(slug, contentRoot, { locale })` returns reviewer-verifiable `resolution` metadata.
- `LocaleAwareContentProjection`, `projectLocaleAwareContent()`, and `projectLocaleAwareContentCatalog()` in `src/lib/content/locale-aware-content-projection.ts` expose durable metadata seams for later navigation and search work; `loadDocPage()` and `projectDocsShellNavigation()` attach `localeProjection` when variant bindings are available.
